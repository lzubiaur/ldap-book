Étude et Implémentation de LDAP Sécurisé

4.9 Contrôle des accès aux données

Sécuriser le trafic réseau entre le client et le serveur LDAP ne sert à rien si l’accès aux données critiques de l’annuaire n’est pas contrôlé. En effet, si n’importe quel utilisateur peut lire les mots de passe enregistrés dans l’annuaire, il est vain de vouloir sécuriser l’authentification des clients LDAP.

Pour gérer les droits d’accès, OpenLDAP implémente un mécanisme d’ACL qui lui est propre. Pour rappel, le standard LDAP ne donne aucune spécification à ce sujet (Chapitre 1 page 20). L’avantage des listes de contrôle d’accès (ACL) de OpenLDAP est une syntaxe simple et puissante.

4.9.1 Syntaxe des ACL

Les ACL de OpenLDAP permettent de répondre à la question qui a accès à quoi? et ont généralement la forme suivante :

access to <what> [by <who> <access> <control>]

Le Quoi ?

Le <what> indique sur quoi s’applique la règle d’accès. Les différentes valeurs que peut prendre ce paramètre sont présentées au tableau 4.6.

quoi? description
* Le caractère "*" représente tous les objets de l’annuaire.
dn.<scopestyle>=<DN> Expression permettant de désigner une entrée ou un sous ensemble de l’annuaire. Le paramètre <scopestyle> peut prendre une des valeurs suivantes : base (l’entrée elle-même), one (les entrées directement en dessous du DN), subtree (le sous ensemble dont le DN est la racine), children (idem subtree mais sans l’entrée DN).
filter=filtreldap Désigne les entrées de l’annuaire après que le filtre soit appliqué.
attrs=liste_attributs Indique la liste des attributs cibles.

Tab. 4.6 – Cibles d’une ACL

Le Qui?

Le <who> identifie les entités pour qui la règle s’applique.

qui? description
* Tout le monde y compris les utilisateurs anonymes et authentifiés.
anonymous Les utilisateurs anonymes (non authentifiés).
users Les utilisateurs authentifiés.
self Le DN de l’entrée cible représente l’utilisateur.
dn=<expression> L’utilisateur désigné par le DN fournit.

Tab. 4.7 – Entités concernées par les ACL

Quels droits ?

Le paramètre <access> représente le droit accordé à l’entité. Le tableau 4.8 reprend chaque niveau d’accès du plus fort au plus faible.

niveau d'accès description
write Droit en écriture.
read Droit en lecture.
search Application de filtre de recherche.
compare Comparaison d’attribut.
auth L’attribut peut être utiliser uniquement à des fin d’authentification.
none Aucun accès.

Tab. 4.8 – Niveaux d’accès d’une ACL

Enfin, le paramètre <control> définit la façon dont les ACL sont évaluées.

niveau d'accès description
stop Valeur par défaut, elle indique que l’évaluation des ACL s’arrête lorsqu’une correspondance est trouvée.
continue Evalue l’ACL en cours et les ACL suivantes.
break Arrête l’évaluation de l’ACL courante mais évalue les ACL suivantes.

4.9.2 Définition des ACL

Les ACL peuvent se trouver soit dans le fichier slapd.conf soit dans un autre fichier. Ce dernier est lié au fichier slapd.conf à l’aide de la clause include.

include "/usr/local/etc/openldap/ldap.access.conf"

Dans un annuaire destiné à contenir des comptes utilisateur, les données à protégées en premier lieu sont les mots de passe. Les autres informations concernant un utilisateur pouvant être connues de tout le monde12, les deux ACL suivantes sont suffisantes.

access to attr=userPassword
       by * auth
access to *
       by * read

La première ACL définit un contrôle d’accès sur l’attribut userPassword de n’importe quelle entrée. L’ACI (Access Control Item) by * auth permet l’utilisation de l’attribut userPassword à des fins d’authentification sans pour autant permettre la lecture de l’attribut. La deuxième ACL autorise la lecture du reste de l’annuaire à tout le monde.

A noter qu’il n’est pas nécessaire de définir les droits de l’utilisateur rootDN définit dans le fichier slapd.conf. En effet, en tant qu’administrateur de l’annuaire, celui-ci n’est pas concerné pas les ACL et possède tout les droits.

4.9.3 Création d’un proxy agent

Dans l’exemple de définition d’ACL donné plus haut, la première ACL restreint l’accès à l’attribut userPassword mais la deuxième ACL autorise n’importe quel utilisateur même anonyme à lire le reste de l’annuaire. Pour éviter que n’importe quelle machine ne puisse accéder anonymement à l’annuaire, il est possible de définir un proxy agent qui servira d’utilisateur mandataire. Ainsi les clients LDAP accéderont à l’annuaire via cet utilisateur et non plus de façon anonyme. L’entrée de type person donnée dans l’exemple suivant peut servir d’utilisateur proxy.

dn: cn=proxyagent,ou=Profile,dc=inpres,dc=be
cn: proxyagent
sn: proxyagent
objectClass: top
objectClass: person
userPassword:: secret

Une fois que cette entrée est ajoutée à l’annuaire, il faut reconsidérer les autorisations d’accès à l’annuaire. Ainsi, l’ACL access to * by * read est modifiée de la sorte :

access to attr=userPassword
       by * auth
access to *
       by dn="cn=proxyagent,ou=Profile,dc=inpres,dc=be" read

Après avoir redémarré le serveur LDAP, seul l’utilisateur cn=proxyagent à le droit de lire tout l’annuaire à l’exception des mots de passe. Le bon fonctionnement du proxy-agent peut être vérifié à l’aide du client ldapsearch.

ldapsearch -x -b dc=inpres,dc=be -D cn=proxyagent,ou=Profile,dc=inpres,dc=be -W

Enter LDAP Password: <mot de passe du proxy-agent>

Si la configuration de l’utilisateur proxy est correcte, les clients LDAP peuvent être mis à jour. Le client Tru64 prévoit deux paramètres (fichier ldapcd.conf) pour indiquer le DN et le mot de passe à utiliser pour se connecter et effectuer les recherches dans l’annuaire.

machine_dn:     "cn=proxyagent,ou=Profile,dc=inpres,dc=be"
machine_pass:   "secret"

Quant aux clients Debian et Solaris, les modules PAM et NSS pour LDAP sont configurés pour se connecter via le proxy agent en mettant à jour les attributs binddn et bindpw du fichier ldap.conf.

binddn   cn=proxyagent,ou=Profile,dc=inpres,dc=be
bindpw   secret

Le mot de passe étant enregistré en clair dans des fichiers accessibles en lecture par tout le monde (0444), l’utilisateur proxy ne doit pas posséder de droit en écriture sur l’annuaire.

12 C’est le cas du fichier /etc/passwd qui est lisible par tous les utilisateurs.
