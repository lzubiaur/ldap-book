Étude et Implémentation de LDAP Sécurisé

4.10 Gestion des mots de passe

La fin de ce chapitre est consacrée à la gestion des mots de passe utilisateur LDAP. Bien que OpenLDAP fournisse la commande ldappasswd, cette dernière n’a pas été conçue pour remplacer la commande système passwd. Il est donc déconseillé d’installer ldappasswd en remplacement de la commande passwd.

Pour changer le mot de passe des utilisateurs locaux, la commande passwd modifie le champ "mot de passe" des entrées du fichier /etc/passwd. Sur des systèmes d’exploitation comme Linux ou Solaris, la commande passwd est configurée pour travailler avec l’API (Application Programming Interface) PAM et est en mesure de modifier des mots de passe LDAP au travers du module pam_ldap.

Pour ce faire, il faut configurer le service passwd dans le fichier /etc/pam.conf sur Solaris et /etc/pam.d/passwd sur Debian (voir Chapitre 3). Dans l’exemple suivant, le service passwd est configuré, dans un premier temps pour réaliser le changement des mots de passe de l’annuaire LDAP (via le module pam_ldap) et en cas d’échec, la modification est effectuée dans la base de données locale (via le module pam_unix).

password   sufficient pam_ldap.so
password   required pam_unix.so nullok obscure min=4 max=8 try_first_pass

Sur le client Tru64, c’est la couche SIA qui sert d’interface entre la commande et le mécanisme de sécurité. Ainsi, la commande passwd fait appel via la fonction siad_chg_password() au mécanisme de sécurité définit dans le fichier de configuration matrix.conf. Dans l’exemple suivant, la commande passwd est configurée pour utiliser le mécanisme BSD (fichiers système locaux) et ensuite le mécanisme de sécurité LDAP.

siad_chg_password=(BSD,libc.so) (LDAP,/usr/shlib/libsialdap.so)

Pour être en mesure de modifier son mot de passe, l’utilisateur LDAP qui fait appel à la commande passwd doit avoir le droit d’écriture sur l’attribut userPassword. Pour ce faire, la règle d’accès concernant l’attribut userPassword doit être adaptée.

access to attr=userPassword
        by self write
        by * auth

Cette ACL permet au client LDAP d’utiliser l’attribut userPassword pour s’authentifier et autorise les clients authentifiés à modifier leur propre mot de passe (by self write). Si un utilisateur proxy est utilisé, l’ACL doit être modifiée. En effet, le mot de passe de l’utilisateur proxy étant connu de tous, tout le monde peut se connecter en tant qu’utilisateur proxyagent et modifier son mot de passe. L’ACL suivante interdit à l’utilisateur proxyagent de modifier son mot de passe.

access to attr=userPassword
       by dn="cn=proxyagent,ou=Profile,dc=inpres,dc=be" auth
       by self write
       by * auth

4.10.1 Chiffrement des mots de passe

Le standard LDAP ne définit pas de mécanisme permettant le chiffrement des données stockées dans l’annuaire. Pourtant il est important de pouvoir chiffrer certaines informations sensibles comme les mots de passe par exemple. OpenLDAP permet le chiffrement de l’attribut userPassorwd. Pour cela, OpenLDAP préfixe la valeur de l’attribut avec un mot clé indiquant l’algorithme de hachage ou de chiffrement utilisé pour encoder la valeur de l’attribut. L’exemple suivant montre l’attribut userPassword utilisant l’algorithme de message digest SHA.

userPassword: {SHA}5en6G6MezRroT3XKqkdPOmY/BfQ=

Le paramètre password-hash du serveur slapd définit l’algorithme à utiliser par défaut pour stocker les mots de passe dans l’attribut userPassword. L’algorithme par défaut est SSHA. Les différents types de chiffrement supportés par le serveur OpenLDAP sont repris ci-dessous.

  • CLEARTEXT :

    Aucun chiffrement n’est appliqué. Le mot de passe est enregistré en clair dans l’annuaire.

  • CRYPT :

    Le mot de passe est chiffré au moyen de la la fonction crypt() de la bibliothèque C standard du système. Ce type de chiffrement permet de rester compatible avec les mots de passe UNIX du fichier passwd.

  • MD5 :

    L’algorithme MD5 produit des digests de 128 bits.

  • SHA :

    L’algorithme SHA développé par NIST (National Institute of Standards and Technology) produit des valeurs de hachage de 160 bits.

  • SSHA :

    L’algorithme de message digest SSHA développé par Netscape offre une plus grande sécurité que l’algorithme SHA-1. L’usage de l’algorithme SSHA pour stocker les mots de passe dans un annuaire LDAP est recommandé.

Lorsque la commande passwd est utilisée pour changer un mot de passe LDAP via le module pam_ldap, plusieurs méthodes de modification sont possibles. Le paramètre pam_password du fichier ldap.conf permet de spécifier une méthode de modification parmi les suivantes :

  • clear : le mot de passe est envoyé en clair au serveur.
  • crypt : le mot de passe est haché à l’aide de la fonction crypt() avant d’être envoyé au serveur.
  • md5 : le digest obtenu avec l’algorithme MD5 est envoyé au serveur.
  • ad : permet de modifier les mots de passe d’un annuaire Active Directory.

Afin de protéger au mieux les mots de passe enregistrés dans l’annuaire, l’algorithme SSHA doit être utilisé. Les clients LDAP utilisant PAM en sont capables. Par contre, le client Tru64 ne supporte que les mots de passe en clair ou chiffrés de type CRYPT. Ce problème de compatibilité oblige l’utilisation du mécanisme CRYPT pour stocker tous les mots de passe.
