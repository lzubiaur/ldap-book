Étude et Implémentation de LDAP Sécurisé

4.7 Configuration du client Tru64 version 5.0A

Jusqu’à présent la configuration du client LDAP sur Tru64 s’est portée sur la version 5.1-B. Cette section est consacrée à la mise en place de LDAP sur la version 5.0A du système d’exploitation HP Tru64 UNIX.

4.7.1 Installation des packages

Le package d’authentification LDAP (OSFLDPAUTH) et des utilitaires LDAP (LDPUTILS) sont disponibles sur le CD-ROM Internet Express Version 6.0. Ce dernier n’étant pas disponible sur le site de HP, plusieurs solutions ont été envisagées.

Installer les packages LDAP de la version supérieur (5.1B)

L’installation des packages LDAP fournis avec la version 5.1B de Tru64 ne fonctionnent pas. La commande setld refuse d’installer des packages d’une autre version de Tru64.

/usr/sbin/mount /dev/disk/cdrom0a /mnt
/usr/sbin/setld -l /mnt/ALPHA/BASE OSFLDPAUTH540
    setld:  You are attempting to install from a distribution media that
        is a different version than your currently installed system.
        You presently have the following operating system version installed:
        Tru64 UNIX V5.0A Operating System ( Rev 1094 )
        Please use a distribution media that matches this version when
        installing base operating system software.

Installer le package W2KSSO100

L’utilisation de LDAP et du démon ldapcd est mentionnée dans la documentation en ligne concernant le package W2KSSO (Windows 2000 Single Sign On). Après l’installation du package et la configuration du fichier ldapcd.conf, les commandes id et finger retrouvent bien les informations concernant les utilisateurs LDAP.

id laurent
uid=10000(laurent) gid=10005(ldap)

finger laurent
Login name: laurent
Directory: /ldap/usr/laurent    Shell: /bin/sh
On since Apr 29 11:51:34        23 seconds Idle Time

Par contre, aucun module LDAP (bibliothèque libsialdap.so) ne permet l’authentification des utilisateur LDAP.

Installation manuelle des bibliothèques

La troisième solution envisagée consiste à copier depuis une machine Tru64 5.1B les fichiers et bibliothèques nécessaires à l’installation de LDAP. Ces fichiers sont repris ci-dessous :

  • le démon ldapcd
  • le fichier de configuration ldapcd.conf
  • les bibliothèques requises par ldapcd
  • le module d’authentification LDAP (bibliothèque libsialdap.so).

L’utilitaire ldd permet d’afficher les bibliothèques dynamiques utilisées par un pro- gramme. Ainsi, la commande ldd appliquée sur le fichier exécutable ldapcd fournit la liste suivante :

ldd /usr/sbin/ldapcd
        Main  =>   /usr/sbin/ldapcd
        libssldap50.so  =>   /usr/shlib/libssldap50.so
        libldap50.so  =>   /usr/shlib/libldap50.so
        libpthread.so  =>   /usr/shlib/libpthread.so
        libexc.so  =>   /usr/shlib/libexc.so
        libc.so  =>   /usr/shlib/libc.so
        libplc4.so  =>   /usr/shlib/libplc4.so
        libplds4.so  =>   /usr/shlib/libplds4.so
        libnspr4.so  =>   /usr/shlib/libnspr4.so
        libnss3.so  =>   /usr/shlib/libnss3.so
        libssl3.so  =>   /usr/shlib/libssl3.so
        libprldap50.so  =>   /usr/shlib/libprldap50.so
        libcxx.so  =>   /usr/shlib/libcxx.so
        librt.so  =>   /usr/shlib/librt.so
        libmach.so  =>   /usr/shlib/libmach.sont

Après avoir retiré les bibliothèques déjà présentes sur le système cible (machine france), la liste exhaustive des fichiers à copier devient :

        /usr/sbin/ldapcd
        /etc/ldapcd.conf
        /usr/shlib/libsialdap.so
        /usr/shlib/libnspr4.so
        /usr/shlib/libprldap50.so
        /usr/shlib/libnss3.so
        /usr/shlib/libplc4.so
        /usr/shlib/libssl3.so
        /usr/shlib/libldap50.so
        /usr/shlib/libplds4.so
        /usr/shlib/libssldap50.so

Si le programme ldapcd ne parvient pas à trouver une bibliothèque, il affiche un message d’erreur et se termine.

/usr/sbin/ldapcd
3048:/usr/sbin/ldapcd: /sbin/loader: Fatal Error: cannot map libldap50.so

4.7.2 Configuration du client

Une fois que les fichiers nécessaires sont copiés sur la machine Tru64 5.0, la configu- ration du client LDAP reste sensiblement la même que pour la version 5.1B.

  1. Configurer le fichier démon ldapcd via le fichier ldapcd.conf.

    directory:      10.59.4.8
searchbase:     "dc=inpres,dc=be"
port:           636
usessl:         1
connections:    6
max_threads:    64
debug:          1

  2. Créer le repertoire /etc/cert7.db et y copier les fichiers cert7.db et key3.db.

  3. Lancer le demon ldapcd et vérifier qu’il a bien établi toutes les connexions avec le serveur LDAP (paramètre connections du fichier ldapcd.conf).

    /usr/sbin/ldapcd

ps x | grep ldap
3029 ??       S        0:00.10 /usr/sbin/ldapcd
3031 pts/2    S  +     0:00.01 grep ldap

/usr/sbin/netstat -na | grep 636
tcp   0  0   10.59.4.1.1987      10.59.4.8.636       ESTABLISHED
tcp   0  0   10.59.4.1.1988      10.59.4.8.636       ESTABLISHED
tcp   0  0   10.59.4.1.1989      10.59.4.8.636       ESTABLISHED
tcp   0  0   10.59.4.1.1990      10.59.4.8.636       ESTABLISHED
tcp   0  0   10.59.4.1.1991      10.59.4.8.636       ESTABLISHED
tcp   0  0   10.59.4.1.1992      10.59.4.8.636       ESTABLISHED

    Le demon ldapcd peut être arrêté proprement en lui envoyant le signal TERM ou le signal INT.

    ps x | grep ldapcd
3281 ??       S        0:00.09 /usr/sbin/ldapcd
3283 pts/2    S  +     0:00.01 grep ldapcd

kill -TERM 3281

  4. Configurer le fichier matrix.conf pour mettre en place l’authentification LDAP et lancer la commande siainit pour initialiser le système.

    # sia matrix configuration file
siad_init=(BSD,libc.so) (LDAP,/usr/shlib/libsialdap.so)
siad_chk_invoker=(BSD,libc.so) (LDAP,/usr/shlib/libsialdap.so)

siad_ses_init=(BSD,libc.so) (LDAP,/usr/shlib/libsialdap.so)
siad_ses_authent=(BSD,libc.so) (LDAP,/usr/shlib/libsialdap.so)

    /usr/sbin/siainit

Successful SIA initialization

  5. Relancer l’environnement graphique CDE au moyen du script xlogin

    /sbin/init.d/xlogin restart

4.7.3 Messages d'erreur SSL

Ce paragraphe reprend les messages d’erreur SSL/TLS rencontrés avec les clients LDAP Tru64 5.1B et 5.0A.

Unknown CA

TLS: can’t accept.
TLS: error:14094418:SSL routines:SSL3_READ_BYTES:tlsv1 alert unknown ca

Le message précédent est une erreur fatale retournée par le sous-protocole SSL Alert chargé de gérer les messages d’erreur. Le message "unknown ca" indique une erreur due à un certificat signé par un CA non reconnu. Dans ce cas, il faut vérifier la présence du certificat du serveur dans le fichier cert7.db du client.

Unknown protocol

TLS: can’t accept.
TLS: error:140760FC:SSL routines:SSL23_GET_CLIENT_HELLO:unknown protocol

Cette erreur est retournée par le sous-protocole SSL Handshake utilisé par le client et le serveur à l’établissement de la session SSL. L’erreur "unknown protocol" (protocole inconnu) est rencontrée à l’envoie du premier message (CLIENT_HELLO). Dans ce cas, il faut vérifier que le client se connecte bien sur le port sécurisé (636) et que SSL soit activé (paramètre usessl à 1).

Expired certificat

TLS: can’t accept.
TLS: error:14094415:SSL routines:SSL3_READ_BYTES:sslv3 alert certificate expired

Le message certificate expired du sous-protocole SSL Alert indique un certifi- cat dont la date de validité est expirée. Si cette erreur apparaît alors que le certificat est valide, il faut accorder l’heure du client sur celle du serveur LDAP (commande date).
